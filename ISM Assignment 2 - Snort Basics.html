Introduction

Snort is a prominent open-source network intrusion detection and prevention system (IDS/IPS) that offers real-time network traffic analysis and packet logging. Developed by Martin Roesch in 1998 and originally released by Sourcefire, Snort employs a rule-based language that combines anomaly detection, protocol analysis, and signature matching techniques to identify potentially malicious network behavior.

Snort detects malicious network traffic by comparing the network packets to a set of rules, often created by Snort and the community. Snort can be used as a packet sniffer, packet logger, and intrusion prevention system.

Snort is available for download and configuration on a range of operating systems, including Linux, Unix, BSD, and Microsoft Windows.

Rule Definition

Snort is a libpcap-based packet sniffer/logger which can be used as a lightweight network intrusion detection system. It features rules-based logging and can perform content searching/matching in addition to detecting a variety of other attacks and probes, such as buffer overflows, stealth port scans, CGI attacks, SMB probes, and much more. Snort has a real-time alerting capability, with alerts being sent to syslog, a separate “alert” file, or even to a Windows computer via Samba

Rule Headers

These define the action to take when any traffic that matches the rule is identified. A rule header consists of five main components:

<action> <protocol> <source IP> <source port> -> <destination IP> <destination port> (options)

  • Action to take: The first component declared in a Snort rule
  • IP addresses: The source and destination
  • Port numbers: The source and destination
  • Direction of traffic:  -> for single direction from source to destination; <> for bidirectional
  • Inspection protocol: “Layer 3” (IP and ICMP) and “Layer 4” (TCP and UDP) protocols

Rule Options

These define the network traffic criteria that need to be met in order for a rule to match, as well as the output when there is a match. Here are some of the rule options available:

  • Message to display when rule matches: Explains the purpose of the rule.
  • Flow state: Specifies the session properties to check for a given packet.
  • Content or pattern: Specifies the content or pattern to find in a packet’s payload or non-payload data.
  • Service or application protocol: Instructs Snort to identify the content or pattern either on the traffic of the specified application-layer service or the source and destination ports specified in the rule header.
  • Snort ID (sid) and revision number (rev): Uniquely identifies a Snort rule or uniquely identifies the revision number of a Snort rule.

Types of Actions

# ALERT
alert tcp any any -> any 80 (msg:"Alert for HTTP 'GET' traffic"; content:"GET"; sid:1000001; rev:1;)

# LOG
log tcp any any -> any 443 (msg:"Log HTTPS 'POST' traffic"; content:"POST"; sid:1000002; rev:1;)

# PASS
pass tcp any any -> any 22 (msg:"Pass SSH traffic"; sid:1000003; rev:1;)

# ACTIVATE
activate tcp any any -> any 80 (msg:"Activate rule for HTTP"; content:"admin"; activates:1000004; sid:1000004; rev:1;)

# DYNAMIC
dynamic tcp any any -> any 80 (msg:"Dynamic rule for HTTP"; sid:1000005; rev:1;)

# DROP
drop tcp any any -> any 21 (msg:"Drop FTP traffic"; sid:1000006; rev:1;)

# REJECT
reject tcp any any -> any 25 (msg:"Reject SMTP traffic"; sid:1000007; rev:1;)

# SDROP
sdrop tcp any any -> any 110 (msg:"Silently drop POP3 traffic"; sid:1000008; rev:1;)

ALERT Examples:

A set of basic Snort rules for HTTP traffic. These rules can help detect common HTTP requests and potential threats.

## HTTP ALERTS

# Detect HTTP GET/POST/PUT/DELETE requests
alert tcp any any -> any 80 (msg:"HTTP GET Request"; content:"GET "; http_method; sid:1000001; rev:1;)
alert tcp any any -> any 80 (msg:"HTTP POST Request"; content:"POST "; http_method; sid:1000002; rev:1;)
alert tcp any any -> any 80 (msg:"HTTP PUT Request"; content:"PUT "; http_method; sid:1000003; rev:1;)
alert tcp any any -> any 80 (msg:"HTTP DELETE Request"; content:"DELETE "; http_method; sid:1000004; rev:1;)

# Detect HTTP User-Agent header
alert tcp any any -> any 80 (msg:"HTTP User-Agent Detected"; content:"User-Agent:"; http_header; sid:1000005; rev:1;)

## OTHER USEFUL ALERTS

# Detect SQL Injection attempt in URL
alert tcp any any -> any 80 (msg:"SQL Injection Attempt"; content:"SELECT "; nocase; http_uri; sid:1000006; rev:1;)

# Detect specific file download (e.g., .exe)
alert tcp any any -> any 80 (msg:"Executable File Download"; content:".exe"; http_uri; sid:1000009; rev:1;)

REJECT Examples:

Reject rules can be useful in various scenarios where you want to actively prevent certain types of traffic and inform the sender of the rejection. Here are some practical use cases for reject rules:

## ACTIVE PREVENTION

# Blocking Unauthorized Access to Critical Ports
reject tcp any any -> 192.168.1.10 22 (msg:"Reject unauthorized SSH access"; sid:1001001; rev:1;)

# Preventing Unauthorized Web Scraping
reject tcp any any -> any 80 (msg:"Reject potential web scraping"; content:"User-Agent: WebScraper"; http_header; sid:1001002; rev:1;)

# Blocking Traffic from Known Malicious IPs
reject ip 192.168.10.100 any -> any any (msg:"Reject traffic from known malicious IP"; sid:1001003; rev:1;)

# Stopping Exploit Attempts
reject tcp any any -> any 443 (msg:"Reject known exploit attempt"; content:"ExploitCode"; sid:1001004; rev:1;)

# Blocking Traffic to Specific Services
reject tcp any any -> any 3306 (msg:"Reject access to MySQL service"; sid:1001005; rev:1;)

# Preventing Access to Restricted Web Pages
reject tcp any any -> any 80 (msg:"Reject access to restricted web page"; content:"/admin"; http_uri; sid:1001006; rev:1;)

# Blocking Traffic from Suspicious User Agents
reject tcp any any -> any 80 (msg:"Reject suspicious user agent"; content:"User-Agent: Bot"; http_header; sid:1001007; rev:1;)

Recommendations

While Snort is a powerful intrusion detection system, it has some limitations. Here are key drawbacks:

  • High false positive rate — Snort often generates false alarms, especially with default rules.
  • Performance issues on high-speed networks — Snort may struggle with traffic on very fast networks.
  • Complex configuration — Setting up and maintaining Snort rules can be challenging.
  • Limited protocol support — Snort may not effectively analyze all modern protocols.
  • Resource-intensive — Snort can consume significant CPU and memory resources.
  • Lack of built-in visualization — Snort's output can be difficult to interpret quickly.
  • Limited encrypted traffic analysis — Snort struggles with inspecting encrypted communications.

Addressing these limitations can help organizations enhance their network security posture while leveraging Snort's strengths in intrusion detection. Here are some strategies:

  • Fine-tune rules, employ machine learning for adaptive rule creation
  • Implement hardware acceleration, use multiple Snort instances in parallel, or consider more scalable alternatives like Suricata.
  • Use rule management tools, improve documentation and training, or consider managed IDS services.
  • Regularly update Snort, use complementary tools for specific protocols, or consider more comprehensive security information and event management (SIEM) solutions.
  • Integrate with visualization tools like ELK stack (Elasticsearch, Logstash, Kibana) or use commercial front-ends for better data representation.
